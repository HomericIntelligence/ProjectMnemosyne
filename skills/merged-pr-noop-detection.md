## Overview

| Field | Value |
| ------- | ------- |
| **Skill** | merged-pr-noop-detection |
| **Category** | tooling |
| **Trigger** | Fix plan file contains "No fixes needed", "PR is merged", or similar no-op signals |
| **Outcome** | Safely exit without making changes, without committing, without running tests |

## When to Use

- A `.claude-review-fix-<issue>.md` file was generated by automation but the plan body says the PR is already merged
- An orchestration script invokes a fix workflow with a stale fix-plan targeting a merged PR
- CI automation creates fix tasks for PRs in `MERGED` or `CLOSED` state
- The fix plan explicitly states "No fixes needed" or "No problems found"

## Verified Workflow

1. **Read the fix plan file** — look for the "Fix Plan:" section in the `.claude-review-fix-*.md` file
2. **Check for no-op indicators** — scan for phrases like:
   - "No fixes needed"
   - "PR is already merged"
   - "No problems found"
   - `gh pr view <N> --json state` returning `"MERGED"`
3. **Confirm PR state via CLI** (optional validation):
   ```bash
   gh pr view <PR_NUMBER> --json state --jq '.state'
   # Returns: MERGED, CLOSED, or OPEN
   ```
4. **Short-circuit** — do NOT:
   - Run tests (`pytest`, `pre-commit`, etc.)
   - Make any code changes
   - Create commits
   - Push branches
5. **Report clearly** — tell the user: "The fix plan indicates PR #N is already merged with no fixes needed. No action taken."

## Failed Attempts

| Attempt | What Was Tried | Why It Failed | Lesson Learned |
| --------- | ---------------- | --------------- | ---------------- |
| Running tests anyway | Executed `pixi run python -m pytest tests/ -v` despite no-op plan | Wastes CI minutes and can produce confusing output suggesting something was "fixed" | Always check fix plan content before executing any workflow steps |
| Creating an empty commit | Committed with "fix: Address review feedback" when there was nothing to fix | Creates misleading history implying work was done | Only commit when actual code changes exist |
| Ignoring the plan content | Skipped reading the fix plan body and just ran all fix steps | Script-generated fix plans may be no-ops when PRs merge between analysis and execution | Fix plans must be read and interpreted before executing any steps |

## Results & Parameters

### Detection Pattern

```python
# Pseudo-code for no-op detection in fix plan files
NO_OP_PHRASES = [
    "No fixes needed",
    "PR is already merged",
    "No problems",
    "no action required",
]

def is_noop_plan(plan_content: str) -> bool:
    return any(phrase.lower() in plan_content.lower() for phrase in NO_OP_PHRASES)
```

### CLI Verification

```bash
# Verify PR state before executing any fix workflow
PR_STATE=$(gh pr view 3189 --json state --jq '.state')
if [ "$PR_STATE" = "MERGED" ]; then
  echo "PR already merged — skipping fix workflow"
  exit 0
fi
```

### Fix Plan File Pattern

Fix plan files in this project follow the naming convention:
`.claude-review-fix-<issue-number>.md`

The file contains a "Fix Plan:" section. If that section's body contains no-op language, the entire workflow should be skipped.

### When NOT to Short-Circuit

- The plan says "MERGED" but lists actual code fixes to apply (backport scenario)
- The plan references open review comments still needing replies
- CI workflows are failing and require fixes even on merged PRs
