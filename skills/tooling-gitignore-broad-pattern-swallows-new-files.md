# Gitignore: Broad Unanchored Directory Pattern Silently Swallows New Source Files

## Overview

| Field | Value |
|-------|-------|
| **Date** | 2026-07-02 |
| **Objective** | Commit a new install smoke-test consumer project under `tests/install/consumer/` in ProjectKeystone (issue #596) — but the freshly written CMake/C++ files never showed up in `git status` |
| **Outcome** | Root cause: the unanchored `.gitignore` entry `install/` (intended only for the repo-root `cmake --install` output tree) matches every directory named `install` at any depth, including `tests/install/`. Fixed with a `!tests/install/` negation placed immediately below the broad pattern; files became trackable and were committed. |
| **Verification** | verified-local (`git check-ignore` rc=1 on the new files; `git status` shows the directory as untracked) |

## When to Use

- Files you just created never appear in `git status` — **not even as untracked**. Suspect an ignore rule, not a tooling bug or a failed write.
- Adding new source/test files whose path shares a name with a common build-output ignore pattern: `install/`, `build/`, `dist/`, `out/`, `target/`.
- Writing a `.gitignore` negation to re-include a directory that a broad unanchored directory pattern excludes.
- An agent/automation workflow produced a commit that is mysteriously missing files, and CI then fails confusingly (e.g. a script references a path that does not exist in the repo).

## Verified Workflow

### Quick Reference

```bash
# 1. Diagnose: which .gitignore file, line, and pattern is responsible?
git check-ignore -v tests/install/consumer/CMakeLists.txt
# → .gitignore:142:install/  tests/install/consumer/CMakeLists.txt

# 2. Fix: add a negation IMMEDIATELY BELOW the broad pattern (re-include the DIRECTORY)
#    .gitignore:
#      # CMake install tree (generated by cmake --install)
#      install/
#      # ... but track the install smoke-test consumer project (issue #596)
#      !tests/install/

# 3. Verify: rc=1 means "not ignored"
git check-ignore tests/install/consumer/CMakeLists.txt; echo rc=$?
git status   # tests/install/ must now appear as untracked
```

### The Trap

An unanchored `.gitignore` entry like `install/` — written to ignore a CMake `cmake --install` output tree at the repo root — matches **every** directory named `install` at **any** depth. When a new, must-be-committed source directory `tests/install/consumer/` (real CMakeLists.txt and C++ sources) was created, the files were written to disk successfully but never appeared in `git status`. Nothing errored anywhere. In an agent/automation workflow this silently ships a commit missing files; CI then fails in a confusing way, e.g. a CI script referencing `tests/install/consumer/` which does not exist in the repository.

### Diagnosis

One command names the exact file, line number, and pattern responsible:

```bash
git check-ignore -v tests/install/consumer/CMakeLists.txt
# .gitignore:142:install/  tests/install/consumer/CMakeLists.txt
```

### Verified Fix

Add a negation **immediately below** the broad pattern, with a comment explaining both entries:

```gitignore
# CMake install tree (generated by cmake --install)
install/
# ... but track the install smoke-test consumer project (issue #596)
!tests/install/
```

Two gitignore subtleties that make this exact form required:

1. **You cannot re-include a file whose parent directory is excluded.** The negation must re-include the *directory itself* (`!tests/install/`). Once the directory is re-included, the files inside it are trackable because nothing else matches them. Negating only the deep files (e.g. `!tests/install/consumer/*`) while the parent stays excluded does nothing.
2. **Order matters.** The negation must come *after* the broad pattern; a later `install/` line would re-exclude it.

Verify:

```bash
git check-ignore tests/install/consumer/CMakeLists.txt; echo rc=$?   # rc=1 → not ignored
git status    # tests/install/ now shown as untracked
```

### Sturdier Alternative

If the ignore is truly root-only, root-anchor the original pattern instead:

```gitignore
/install/
```

Prefer the narrow negation, however, when you cannot audit every deep `install/` directory the broad pattern may be intentionally covering — root-anchoring changes behavior for *all* nested `install/` dirs at once.

### Process Rule

After **creating** new files in a repo, always confirm they appear in `git status` before finishing. Absence from status is a signal, not a success — the Write succeeded, but git will never see the file.

**Context:** ProjectKeystone issue #596 (install smoke-test CI check). New consumer project at `tests/install/consumer/` collided with the pre-existing `.gitignore` entry `install/`.

**Working `.gitignore` block:**

```gitignore
# CMake install tree (generated by cmake --install)
install/
```

**Verification commands and expected results:**

| Command | Expected |
|---------|----------|
