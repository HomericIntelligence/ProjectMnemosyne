## What Happened

An automated review-fix script generated a fix plan file at:
`.claude-review-fix-3084.md`

The file asked Claude to implement fixes for PR #3189 (issue #3084). However, when the
fix plan content was read, it was clear that:

1. PR #3189 was already **MERGED** to main
2. All 32/32 CI test groups passed
3. There were no open review comments
4. The plan explicitly stated: "No fixes needed — PR is already merged"

The fix plan still included post-fix instructions:
- Run pytest
- Run pre-commit
- Fix any issues found
- Commit all changes

These instructions are generated generically by the automation script and do not
account for the no-op case.

## Fix Plan File Location

`.claude-review-fix-3084.md` — generated by automation, living in the worktree root.

## Key Observation

The script that generates fix plan files does not differentiate between:
- PRs with actual issues needing fixes
- PRs that are already merged with no open issues

The plan analysis phase produces a no-op result but the template still wraps it
in "implement fixes" instructions. This causes confusion when Claude receives the
file and must determine whether to act or not.

## Resolution

Recognized the no-op signal in the plan body ("No fixes needed — PR is already merged")
and correctly short-circuited without:
- Running tests
- Making code changes
- Creating commits

## Root Cause

The fix-plan generation script does not gate the "implement fixes" wrapper on whether
the plan analysis found any actual issues. This is a script-level gap to address.
